# [V2.2] Workspace-scoped Access Context — Header-based contract

**Repo destino**: `avanzza-core-api` (NestJS backend)
**Origen del request**: `avanzza` (Flutter app — Access subsystem V2.2)
**Estado**: Proposal — pendiente confirmación backend
**Owner Flutter**: equipo Avanzza app
**Owner Backend**: TBD
**Bloquea**: V2.2 client (`AccessActionPolicy`, `canPerform`, CoordinationFlow gates)

---

## TL;DR

Migrar el contrato de Access Context del cliente Flutter a un modelo **stateless multi-workspace** basado en header obligatorio `X-Workspace-Id` que transporta **`Workspace.id` (UUID canónico)**, no la cadena legacy `orgId`. Eliminar dependencia del JWT claim como fuente única de tenancy operativa. JWT pasa a ser **identidad pura**.

Este contrato:

- Habilita per-workspace context refresh sin estado servidor.
- Cierra una grieta de tenancy implícita ("READ vs WRITE workspace context" desalineados).
- Resuelve la ambigüedad `workspaceId` ↔ `orgId` en el wire (header siempre canónico).
- Es prerrequisito para `CoordinationFlow` gates server-side seguros (V2.2 + V3).
- Es **aditivo y migrable**: período de deprecación durante el cual el header es opcional + fallback a JWT claim.

## Glosario semántico (lectura obligatoria antes del contrato)

Avanzza tiene **dos identificadores de workspace** que el contrato NO debe mezclar:

| Identificador | Tipo | Origen | Uso permitido |
|---|---|---|---|
| `Workspace.id` | UUID v4 canónico | Primary key de la tabla `Workspace` en backend | **Header `X-Workspace-Id`**, joins canónicos, autoridad final |
| `orgId` (`Workspace.orgId`) | String legacy bridge | Campo separado en `Workspace`, mapeo a partition keys de Firestore/Isar y al modelo previo | Particionamiento de datos en colecciones legacy, scoping de queries de negocio |

**Reglas no-negociables del wire:**

1. El header `X-Workspace-Id` SOLO acepta `Workspace.id` (UUID canónico).
2. La response distingue **explícitamente** `workspaceId` (canonical) y `orgId` (bridge legacy) como campos separados — nunca uno como alias del otro.
3. El backend NUNCA debe aceptar `orgId` legacy en el header. Requests con bridge string en lugar de UUID → 400 `INVALID_WORKSPACE_ID_FORMAT`.
4. El cliente Flutter podrá seguir persistiendo internamente ambos identificadores en su `AccessContextSnapshot` para soporte legacy, pero el wire al backend SIEMPRE usa el canónico.

## Background

### Estado actual

Hoy `GET /v1/access/me/context` resuelve el workspace activo desde el JWT claim `activeContext.organizationId`. Esto funciona para usuarios con un único workspace activo, pero introduce dos problemas en el modelo Avanzza:

1. **Multi-workspace silenciado.** Un usuario con N memberships activas solo puede consultar el contexto de **uno** (el del JWT claim). El cliente no puede pre-cargar caché para los otros, ni validar capabilities cruzadas (ej. validar que actúa como provider en WS-B mientras opera asset-admin en WS-A).

2. **Tenancy implícita en mutaciones.** Las mutaciones (`POST /purchase-request`, etc.) heredan tenancy del JWT claim. Si el cliente quiere "previsualizar" otro workspace sin cambiar su tenancy operativa, no hay mecanismo seguro: cualquier mutación accidental va al workspace del claim.

### Cliente Avanzza V2.1 — ya en producción

Flutter ya implementó `AccessContextSnapshot` persistido en Isar con composite key `(userId, workspaceId)`. Ver [V2.1 docs internos del cliente](../../README.md). El snapshot soporta multi-workspace por diseño.

Lo único que falta: **el endpoint de Core API que el snapshot consume**. Hoy resuelve solo el workspace del JWT. V2.2 necesita resolver workspaces explícitos.

### Por qué stateless (no switch endpoint)

La opción "POST `/workspaces/switch` + GET `/context`" fue evaluada y **descartada** porque:

- Resucita el patrón de la Cloud Function `switchActiveOrganization` que el equipo backend **ya eliminó** conscientemente.
- Introduce session affinity o re-emisión de JWT (tabla de session o pipeline de claims).
- No escala a "refrescar N workspaces" — cada switch invalida el contexto previo.
- Race conditions cross-device.
- Incompatible con CoordinationFlow (validar permisos de target workspace sin abandonar el operativo).

## Decision

| # | Decisión |
|---|---|
| 1 | **Header obligatorio** `X-Workspace-Id` en todas las requests. Después del período de deprecación es **mandatorio**; missing → 400. |
| 2 | **JWT** autentica identidad (uid, email, claims de auth provider). **NO** define contexto operativo de workspace. |
| 3 | **`/v1/access/me/context`** lee el workspace del header (no query param). Retorna contexto de ese workspace o error si la membership no es válida. |
| 4 | **Toda mutación** debe enviar el mismo header `X-Workspace-Id` y el server valida `MembershipPolicy.assertActive(uid, workspaceId)` antes de procesar. |
| 5 | **Leer contexto de WS-B no habilita actuar en WS-B.** Para actuar en B, la mutación debe enviar header=B + el guard de membership lo valida. Self-enforcing. |
| 6 | **`versionTag`** se emite en cada response y debe incluir `providerProfileId` y `providerWorkspaceId` además de los campos canónicos. Permite al cliente detectar drift sin comparar capabilities[] elemento a elemento. |
| 7 | **Limitación documentada**: revocaciones de permisos en backend NO se propagan en tiempo real a clientes activos. Cada cliente refresca al siguiente cold start o al cumplir staleness threshold (24h). Acciones networkCritical sobre snapshots stale fallarán con 403, lo cual es esperado. **V3** resolverá con FCM push invalidation (out of scope V2.2). |

## Contract Specification

### 1. Header `X-Workspace-Id`

| Campo | Spec |
|---|---|
| **Nombre** | `X-Workspace-Id` |
| **Tipo de valor** | UUID v4 — **`Workspace.id` canónico**. NO acepta `orgId` legacy bridge. |
| **Formato** | RFC 4122 lowercase con guiones (8-4-4-4-12) |
| **Aplicabilidad** | TODAS las requests autenticadas a `/v1/*` excepto las que se documenten explícitamente como workspace-agnostic (login, password reset, etc.) |
| **Período de deprecación** | Opcional con warning durante 30 días desde release de V2.2 backend. Mandatorio después. |
| **Ausente (Phase 1)** | Backend resuelve fallback a JWT claim `activeContext.organizationId` (comportamiento actual). Loggea warning. |
| **Ausente (Phase 2)** | Backend responde 400 `MISSING_WORKSPACE_HEADER`. |
| **Inválido (UUID malformado)** | Backend responde 400 `INVALID_WORKSPACE_ID_FORMAT`. |
| **Valor es legacy `orgId` bridge en lugar de `Workspace.id`** | Backend responde 400 `INVALID_WORKSPACE_ID_FORMAT` con detail "header value must be Workspace.id, not legacy orgId". |
| **Workspace no existe** | Backend responde 404 `WORKSPACE_NOT_FOUND` (consistente entre endpoints). |
| **Sin membership ACTIVE** | Backend responde 403 `MEMBERSHIP_NOT_FOUND_OR_INACTIVE`. |

**Ejemplo:**
```http
GET /v1/access/me/context HTTP/1.1
Host: api.avanzza.com
Authorization: Bearer <Firebase ID token>
X-Workspace-Id: 4c94fb65-b556-45a7-8735-c38681e8fd13
```

### 2. `GET /v1/access/me/context`

#### Request

```http
GET /v1/access/me/context HTTP/1.1
Authorization: Bearer <Firebase ID token>
X-Workspace-Id: <UUID>
```

- **Sin** query params nuevos.
- Sin body.
- El workspace a resolver viene **exclusivamente** del header.

#### Response 200 (success)

```json
{
  "isProvisioned": true,
  "user": {
    "id": "uuid",
    "status": "ACTIVE"
  },
  "workspaceId": "uuid-canónico-resuelto",
  "activeWorkspace": {
    "id": "uuid-canónico",
    "orgId": "legacy-bridge-string",
    "state": "ACTIVE"
  },
  "activeOrgId": "legacy-bridge-string",
  "membership": {
    "id": "uuid",
    "status": "ACTIVE",
    "role": "OWNER"
  },
  "capabilities": [
    "purchase_request.read",
    "purchase_request.create",
    "provider.create",
    "provider.read"
  ],
  "providerProfileId": "uuid-or-null",
  "providerWorkspaceId": "uuid-or-null",
  "requiresAction": "NONE",
  "requiresTokenRefresh": false,
  "versionTag": "sha256:e3b0c44298fc1c149afbf4c8996fb924..."
}
```

**Distinción explícita de identificadores en la response:**

| Campo | Tipo | Significado | Uso cliente |
|---|---|---|---|
| `workspaceId` (top-level) | UUID canónico | `Workspace.id` resuelto del header. **Source of truth canónico**. | Persistir como identificador canónico del snapshot. Header de mutaciones futuras. |
| `activeWorkspace.id` | UUID canónico | Mismo valor que `workspaceId`. Nest preservado por compat. | Igual a `workspaceId`. |
| `activeWorkspace.orgId` | String legacy bridge | `Workspace.orgId` — clave de partición para datos legacy en Firestore/Isar. | Scope de queries de negocio (assets, purchases) en colecciones legacy. |
| `activeOrgId` (top-level) | String legacy bridge | Mismo valor que `activeWorkspace.orgId`. Mantenido por compat con clientes pre-V2.2. **Marcado como legacy en este contrato**. | Cliente nuevo: ignorar. Cliente legacy: sigue funcionando. |

**Cambios respecto al contrato actual:**
- **Nuevo top-level `workspaceId`**: el UUID canónico resuelto. Es el identificador que el cliente debe pasar a futuros headers `X-Workspace-Id`.
- `activeOrgId` se mantiene en la response para backwards compat, pero conceptualmente representa **el bridge legacy**, no el "workspace activo canónico".
- Nuevos campos top-level: `providerProfileId`, `providerWorkspaceId`, `versionTag`.
- `membership.role` es nuevo (string canónico: `OWNER` | `ADMIN` | `MEMBER` | `PROVIDER_AGENT` | etc.).

#### Response — `requiresAction` distinto de `NONE`

Mantiene el contrato actual:

| `requiresAction` | Significado | Cuándo |
|---|---|---|
| `NONE` | Caller listo para operar en este workspace | Membership ACTIVE + Workspace ACTIVE + User ACTIVE |
| `BOOTSTRAP_REQUIRED` | Usuario aún no provisioned en Postgres | Primer login, requiere `POST /v1/auth/bootstrap` |
| `SELECT_WORKSPACE` | El header refiere a workspace inválido para este user | Header presente pero no es workspace activo del user |
| `CONTACT_SUPPORT` | Policy blocker terminal | User SUSPENDED/INACTIVE/PENDING |

### 3. Generación de `versionTag`

#### Algoritmo

```
versionTag = "sha256:" + sha256_hex(canonicalJson(payload))

donde payload = {
  userId,                   // UUID del User
  workspaceId,              // Workspace.id CANÓNICO (UUID). NO usar orgId bridge.
  membershipId,             // UUID
  role,                     // string canónico (OWNER/ADMIN/...)
  membershipStatus,         // ACTIVE/INACTIVE
  workspaceState,           // ACTIVE/INACTIVE/ARCHIVED
  capabilities: capabilities.sorted(),
  providerProfileId,        // UUID o null — incluido aunque sea null
  providerWorkspaceId       // Workspace.id CANÓNICO o null
}
```

**Importante:** el `workspaceId` y `providerWorkspaceId` del payload son los **UUID canónicos**, no los `orgId` legacy. Si se mezclan, dos snapshots con el mismo workspace producirán hashes distintos según qué identificador se use, rompiendo la deduplicación cliente.

#### Reglas obligatorias

1. **Canonical JSON**: claves ordenadas alfabéticamente, sin whitespace, valores `null` incluidos explícitamente.
2. **`capabilities.sorted()`**: ordenamiento lexicográfico ASCII antes de hashear. Sin esto, dos backends podrían emitir hashes distintos para el mismo set.
3. **Determinista entre instancias**: el mismo payload canonical debe producir el mismo hash en cualquier réplica del backend.
4. **Cliente lo trata como opaque**: el cliente NO inspecciona el contenido del hash. Solo compara `previous.versionTag == response.versionTag`.

#### Tests obligatorios backend

```
- versionTag determinismo: 100 ejecuciones con mismo payload → mismo hash.
- versionTag sensibilidad: cambiar 1 campo del payload → hash distinto.
- versionTag entre instancias: ejecutar en N réplicas paralelas → mismo hash.
- versionTag con providerProfileId distinto pero capabilities iguales → hash distinto.
```

### 4. Errores

| HTTP | `body.code` | Cuándo | Comportamiento esperado del cliente |
|---|---|---|---|
| **400** | `MISSING_WORKSPACE_HEADER` | Header ausente (post-grace period) | Cliente bug — log + force header en futuro request |
| **400** | `INVALID_WORKSPACE_ID_FORMAT` | Valor del header no parsea como UUID válido. Cubre: cadena no-UUID, formato malformado, valor que parece bridge legacy `orgId` en lugar de `Workspace.id` canónico. | Cliente bug — no retry, log error con valor recibido enmascarado para debugging |
| **401** | `UNAUTHENTICATED` | JWT missing/invalid/expirado | Force re-auth (Firebase) |
| **401** | `USER_NOT_PROVISIONED` | uid no existe en Postgres | Trigger BOOTSTRAP_REQUIRED flow (mantenido) |
| **403** | `WORKSPACE_ACCESS_FORBIDDEN` | Workspace existe y está ACTIVE, pero el uid **NO tiene NINGUNA membership** (ni siquiera INACTIVE) en él. Acceso jamás concedido. | `clearForWorkspace(uid, wsId)` + redirigir a selector. **Loggear como evento de seguridad** (intento de acceso a workspace ajeno). |
| **403** | `MEMBERSHIP_NOT_FOUND_OR_INACTIVE` | uid TIENE membership en el workspace pero su status NO es ACTIVE (suspended, inactive, left, etc.). Acceso revocado o pendiente. | `clearForWorkspace(uid, wsId)` + redirigir a selector + sugerir re-onboarding/contacto soporte según el caso. |
| **403** | `USER_SUSPENDED` | Policy blocker (user nivel) | `clearForUser(uid)` + state=Blocked + overlay |
| **403** | `USER_INACTIVE` | Policy blocker (user nivel) | Idem `USER_SUSPENDED` |
| **403** | `USER_PENDING_ACTIVATION` | Policy blocker (user nivel) | Idem |
| **403** | `WORKSPACE_NOT_ACTIVE` | Workspace existe pero está INACTIVE/ARCHIVED | `clearForWorkspace` + redirigir a selector |
| **404** | `WORKSPACE_NOT_FOUND` | UUID del header no existe | `clearForWorkspace` + redirigir a selector |
| **5xx** | — | Backend error / red caída | `markSyncFailed` en cliente, **no** invalidar snapshot, mantener cache local-first |

**Regla canónica de invalidación cliente:**
- 401 → re-auth (no invalida snapshots — un re-auth exitoso los preserva).
- 403 `WORKSPACE_ACCESS_FORBIDDEN` → `clearForWorkspace(uid, wsId)` + log seguridad.
- 403 `MEMBERSHIP_NOT_FOUND_OR_INACTIVE` → `clearForWorkspace(uid, wsId)` + sugerir re-onboarding.
- 404 sobre workspace específico → `clearForWorkspace(uid, wsId)`.
- 403 user-level (`USER_SUSPENDED` etc.) → `clearForUser(uid)` (todos los workspaces).
- 5xx/red → `markSyncFailed` (NO clear, mantener cache).

### 4.5 Observability obligatoria backend

Toda resolución de `/v1/access/me/context` y toda mutación workspace-scoped DEBE emitir un log estructurado con el siguiente prefijo y campos mínimos:

```
[ACCESS_CONTEXT] uid=<masked-uid> workspaceId=<uuid> membership=<status> role=<role> result=<200|403|...> code=<error-code-or-none>
```

**Campos:**

| Campo | Tipo | Notas |
|---|---|---|
| `uid` | string enmascarado | Solo primeros 4 chars + `***` (e.g. `MPcb***`). NUNCA loggear uid completo. |
| `workspaceId` | UUID canónico | Del header `X-Workspace-Id` (o JWT claim si Phase 0). |
| `membership` | string | `ACTIVE` \| `INACTIVE` \| `NONE` \| `INVALID_WORKSPACE` |
| `role` | string \| `none` | `OWNER` \| `ADMIN` \| ... \| `none` cuando no aplica |
| `result` | int | HTTP status |
| `code` | string \| `none` | `body.code` cuando es error; `none` en 200 |

**Por qué obligatorio**: sin este log, debugging de bugs multi-actor (ej. "el user actuó en el workspace equivocado", "permisos no se aplicaron") es prácticamente imposible. El log es el único punto donde se cruza identidad + workspace + capabilities resueltas en un mismo registro.

**Ejemplo:**

```
[ACCESS_CONTEXT] uid=MPcb*** workspaceId=4c94fb65-b556-45a7-8735-c38681e8fd13 membership=ACTIVE role=OWNER result=200 code=none
[ACCESS_CONTEXT] uid=A1b2*** workspaceId=ff00ee11-... membership=NONE role=none result=403 code=WORKSPACE_ACCESS_FORBIDDEN
[ACCESS_CONTEXT] uid=MPcb*** workspaceId=4c94fb65-... membership=INACTIVE role=ADMIN result=403 code=MEMBERSHIP_NOT_FOUND_OR_INACTIVE
```

**Eventos que NO loggear** (nivel debug solamente, NO info):
- 5xx por red caída (ruido).
- 401 expirado normal (esperado en sesiones largas).

### 5. Comportamiento across mutations

Toda mutación a `/v1/*` workspace-scoped DEBE enviar `X-Workspace-Id` y pasar el guard `MembershipPolicy.assertActive(uid, workspaceId)` antes de procesar. El header es la única fuente de tenancy operativa; el JWT solo autentica.

Como el header es obligatorio en cada request, la regla "leer contexto de un workspace no habilita actuar en él" es **self-enforcing**: para actuar en B, la mutación debe llevar header=B + el guard valida membership en B. No hay vía de escape.

#### Ejemplo de flujo

```http
# Lectura de contexto de WS-B (no muta nada, solo capabilities)
GET /v1/access/me/context
X-Workspace-Id: B
→ 200 OK con capabilities de B

# Mutación en WS-A (header explícito, validado contra membership en A)
POST /v1/purchase-requests
X-Workspace-Id: A
{ ... }
→ guard valida membership(uid, A) → procesa en A

# Mutación en WS-B sin membership en B
POST /v1/purchase-requests
X-Workspace-Id: B
→ 403 WORKSPACE_ACCESS_FORBIDDEN (jamás tuvo membership)
  o 403 MEMBERSHIP_NOT_FOUND_OR_INACTIVE (membership existe pero no ACTIVE)
```

## Migration Plan

### Phase 0 — Backend prepara soporte (Sprint 1)

- [ ] Backend agrega lectura de `X-Workspace-Id` en interceptor/middleware global.
- [ ] Si header presente: valida UUID + `MembershipPolicy.assertActive` + usa el header como source of truth de workspace.
- [ ] Si header ausente: fallback al JWT claim `activeContext.organizationId` (comportamiento actual). **Loggear warning** con `requestPath` + `uid` para tracking.
- [ ] `/v1/access/me/context` retorna nuevos campos: `providerProfileId`, `providerWorkspaceId`, `versionTag`.
- [ ] `membership.role` se emite en response.
- [ ] Generación de `versionTag` con tests de determinismo.
- [ ] Métrica/dashboard de "% de requests sin header" por endpoint para tracking de adopción.

**Criterio de release**: ningún cliente legacy en campo se rompe. Comportamiento sin header = comportamiento actual.

### Phase 1 — Cliente Flutter envía header (Sprint 2)

**Nota crítica de mapping**: hoy el cliente Flutter usa `UserModel.activeContext.orgId` como identificador del workspace activo. Ese campo se llama `orgId` por razones legacy del modelo Isar/Firestore, pero su VALOR ya es UUID v4 (mismo `Organization.id` generado en `CompleteOnboardingUC`). En la mayoría de instalaciones existentes ese UUID **coincide** con `Workspace.id` canónico del backend (porque el backend lo genera o lo recibe de bootstrap). Sin embargo, hay casos posibles de divergencia si el backend persiste un `Workspace.id` distinto al `orgId` que el cliente conoce.

Tareas obligatorias:

- [ ] **Auditoría de mapping cliente↔backend**: confirmar para 100% de los workspaces existentes en producción que `client.activeContext.orgId == backend.Workspace.id`. Reporte explícito antes de Phase 1 release.
- [ ] **Si hay divergencia detectada**: cliente persiste AMBOS identificadores en `AccessContextSnapshot` (agregar campo `canonicalWorkspaceId` opcional al modelo Isar) y resuelve el canónico desde el primer response de `/access/me/context` (que devuelve `workspaceId` top-level). El header se envía con el canónico, no con `activeContext.orgId`.
- [ ] **Si no hay divergencia**: cliente envía `activeContext.orgId` directamente como `X-Workspace-Id`. El backend lo recibe como UUID válido y todo funciona. Documentar la equivalencia confirmada.
- [ ] Dio interceptor en cliente Flutter agrega `X-Workspace-Id` automáticamente con el UUID canónico (resolución según los dos casos anteriores).
- [ ] Endpoint `/v1/access/me/context` consumido con header obligatorio.
- [ ] Snapshot service consume nuevos campos: `workspaceId` (canónico, top-level), `versionTag`, `providerProfileId`, `providerWorkspaceId`.
- [ ] Snapshot service almacena `workspaceId` canónico en el composite key (no el bridge legacy).
- [ ] **Backfill de snapshots existentes** (si hay divergencia): primer cold start tras upgrade detecta snapshots V2.1 con bridge en `workspaceId` y los reemplaza con el canónico tras consultar `/access/me/context` exitosamente.
- [ ] Tests de integración cliente-backend que verifican el round-trip de identificadores.

**Criterio de release**: 100% de las requests del cliente Flutter envían el header con UUID canónico (no con bridge legacy). Reporte de auditoría adjunto.

### Phase 2 — Backend hace header mandatorio (Sprint 3, después de 30 días)

- [ ] Backend rechaza requests sin `X-Workspace-Id` con `400 MISSING_WORKSPACE_HEADER` (para endpoints workspace-scoped).
- [ ] Whitelisting de endpoints workspace-agnostic (login, etc.) documentado.
- [ ] Fallback a JWT claim **eliminado**.
- [ ] Métrica `access_context.header_present_rate` ≥ **99%** sostenido durante 7 días consecutivos.

**Criterio de release explícito**: el cutoff a header obligatorio NO se ejecuta hasta que la métrica de adopción cruce el threshold del 99% durante la ventana de 7 días. Si el ratio se estanca debajo del 99%, ops debe identificar el origen del 1%+ residual (clientes legacy en campo, integraciones server-to-server, scripts) y coordinar su migración antes del cutoff.

#### Métrica de gate

```
Metric:    access_context.header_present_rate
Tipo:      gauge (porcentaje 0.0 - 1.0) o ratio counter
Cálculo:   count(requests_with_X-Workspace-Id) / count(total_workspace_scoped_requests)
Ventana:   ventana móvil de 7 días
Gate Phase 2: > 0.99 sostenido
Alerta:    si baja del 99% post-cutoff (regresión), pageable
```

**Métricas auxiliares recomendadas** (para diagnóstico cuando el gate no se cruza):

```
access_context.header_canonical_rate
  → de los requests con header, % que envían UUID canónico vs bridge legacy.
  → si esta métrica es baja, hay clientes que mandan orgId legacy → 400 INVALID_WORKSPACE_ID_FORMAT.

access_context.header_missing_by_endpoint
  → breakdown por endpoint del 1%+ residual sin header.
  → identifica si el 1% es un endpoint específico o ruido distribuido.
```

### Phase 3 — Cleanup (sprint posterior)

- [ ] Documentación de `JWT.activeContext.organizationId` actualizada para indicar que es legacy y NO se usa para tenancy.
- [ ] Eventual: dejar de emitir el claim en JWT (si Cloud Function lo emitía). Solo emitir identidad pura.

## Acceptance Criteria

### Backend (`avanzza-core-api`)

- [ ] `GET /v1/access/me/context` acepta `X-Workspace-Id` y resuelve contexto de ese workspace.
- [ ] Header solo acepta `Workspace.id` (UUID canónico). Si recibe legacy `orgId` (bridge string) → 400 `INVALID_WORKSPACE_ID_FORMAT` con detail explícito.
- [ ] Sin header (Phase 0): fallback al JWT claim + warning loggeado con `requestPath` + `uid`.
- [ ] Header con UUID malformado → 400 `INVALID_WORKSPACE_ID_FORMAT`.
- [ ] Header con workspace inexistente → 404 `WORKSPACE_NOT_FOUND`.
- [ ] Header con workspace donde uid **no tiene NINGUNA membership** → 403 `WORKSPACE_ACCESS_FORBIDDEN` (loggeado como evento de seguridad).
- [ ] Header con workspace donde uid tiene membership pero NO ACTIVE → 403 `MEMBERSHIP_NOT_FOUND_OR_INACTIVE`.
- [ ] `WORKSPACE_ACCESS_FORBIDDEN` y `MEMBERSHIP_NOT_FOUND_OR_INACTIVE` son códigos distintos y testeados por separado.
- [ ] Header con workspace INACTIVE/ARCHIVED → 403 `WORKSPACE_NOT_ACTIVE`.
- [ ] User SUSPENDED/INACTIVE/PENDING → 403 con código específico.
- [ ] **Response incluye `workspaceId` top-level (UUID canónico)** además de `activeOrgId` (bridge legacy).
- [ ] Response incluye `providerProfileId`, `providerWorkspaceId` (UUID canónico), `versionTag`, `membership.role`.
- [ ] `workspaceId` y `activeWorkspace.id` siempre coinciden (mismo UUID canónico).
- [ ] `activeOrgId` y `activeWorkspace.orgId` siempre coinciden (mismo bridge legacy).
- [ ] `versionTag` determinista entre instancias (test paralelo con N replicas).
- [ ] `versionTag` sensible a cambios en cualquiera de los 9 campos del payload.
- [ ] `versionTag` payload usa `workspaceId` canónico, NO `orgId` bridge.
- [ ] Mutaciones (al menos `POST /v1/purchase-requests`) validan `X-Workspace-Id` con guard de membership y rechazan bridge legacy.
- [ ] **Logging estructurado** `[ACCESS_CONTEXT]` emitido en cada `/v1/access/me/context` y mutaciones workspace-scoped (uid enmascarado, workspaceId, membership status, role, result, code).
- [ ] Métrica `access_context.header_present_rate` expuesta (ventana móvil 7 días). Gate Phase 2: > 99% sostenido durante 7 días consecutivos antes de hacer header obligatorio.
- [ ] Métricas auxiliares: `access_context.header_canonical_rate`, `access_context.header_missing_by_endpoint`.

## Limitación documentada (importante)

> **Revocaciones cross-device no se propagan en tiempo real.**
>
> Si un admin revoca permisos a un usuario, los clientes ya activos del usuario **mantienen su snapshot local** hasta que ocurra una de:
> - Próximo cold start del cliente (snapshot expira o se refresca).
> - Acción networkCritical que el cliente intenta — el server responde 403 y el cliente invalida el snapshot.
> - Stale threshold (24h fresh / 72h critically stale) dispara warning en UI.
>
> **Mitigación V2.2 (cliente):** `canPerform(networkCriticalAction, snapshot)` exige `syncStatus=confirmed && isFresh`. Acciones networkCritical NO se ejecutan offline o con snapshot stale.
>
> **Solución V3:** server emite evento FCM al topic `access:revoked:<uid>` cuando hay revocación. Cliente subscribe y dispara `clearForUser` inmediato. Out of scope V2.2.
